7. Interpolation Animations

META Transforms are cool and in the majority of the cases they are great! However, they do not provide a lot of control. We'll talk about how Lerp, and Keyframe give us more precision and let us make really cool animations.

7.1 Functor Interpolation

DEFINITION For two functors of the same type, functor interpolation is defined to be the recursive interpolation of its arguments. That is, a new functor is created of the same type where each argument to the functor is just the interpolation from the associated argument of the start functor to that of the end functor.

Here's an example:

var A = Circle:
    center: {0,0,0}
    radius: 1
    tag: {}
    stroke: BLACK

var B = Circle:
    center: {1,0,0}
    radius: 3
    tag: {}
    stroke: BLACK

var C = lerp(A, B, 0.25)

In this case, this is the same thing as saying:

C = Circle:
    center: {0.25, 0, 0}
    radius: 1.5
    tag: {}
    stroke: BLACK

Because the center is interpolated from {0,0,0} to {1,0,0}, and the radius is interpolated from 1 to 3.

For full semantics, read the documentation of lerp.

7.2 Lerp

Now, what if we can smoothly interpolate between two meshes using functor interpolation? This might sound similar to Transform, but note that Transform works by matching points and thus is slightly different in semantics.

IMPORTANT This brings us to Lerp. It does exactly what we want, interpolating a screen variable between two functors. In particular, at each time t, Lerp sets its target variable equal to the functor interpolation between the start and end, with the blend factor being time dependent.

Heres an example. 

tree trans
tree lerps
let trans_square = Square:
    center: RIGHT
    width: 0.5
    tag: {}
    stroke: BLACK

let lerp_square = Square:
    center: LEFT
    width: 0.5
    tag: {}
    stroke: BLACK

play += Show:
    into&: trans
    value: trans_square
    sticky: 0
play += Show:
    into&: lerps
    value: lerp_square
    sticky: 1

let trans_rot = Rotated:
    mesh: trans_square
    rotation: 5 * PI / 4

play += Transform:
    mesh&: trans
    target: trans_rot
    null_action: fade
    time: 1
    sticky: 0

/* make sure arguments to lerp are the same functor! */
let lerp0 = Rotated:
    mesh: lerp_square
    rotation: 0

let lerp1 = Rotated:
    mesh: lerp_square
    rotation: 5 * PI / 4

play += Lerp:
    dst&: lerps
    start: lerp0
    end: lerp1
    time: 1
    sticky: 1

Notice how Transform takes the shortest path, whereas Lerp performs an actual rotation since we are interpolating the rotation factor.

REMARK The reason we have to specify the starting value (which seems like we can just derive it from the current state) is a bit complex, but essentially whenever we show meshes, we actually do lose information about the function it came from and that needs to be provided again.

7.3 User Defined Functors and More Examples

The true power of Lerp comes when we use user defined types.

For instance, here we interpolate several factors of the Weirstrass function. In general, Lerp is good for demonstrating how several parameters influence an object.

let n = 50

func ws(a,b,x) = identity:
    var s = 0
    for i in 0:<n
        s += a ** i * cos((b ** i) * PI * x)
    element: s

func Weierstrass(a,b) = ExplicitFunc:
    start: -4
    stop: 4
    samples: 2048
    f(x): ws(a,b,x)
    tag: {}
    stroke: RED

tree main
tree static

let w = Weierstrass:
    a: 0.5
    b: 0.1
let axis = Axis2d:
    center: ORIGIN
    x_unit: 1
    x_rad: 4
    x_label: "x"
    y_unit: 1
    y_rad: 3
    y_label: "y"
    grid: off
    tag: {}
    color: BLACK

play += Show:
    into&: main
    value: w
    sticky: 0
play += Show:
    into&: static
    value: axis
    sticky: 0
play += Lerp:
    var next = w
    next.b = 4
    dst&: main
    start: w
    end: next
    time: 3
    unit_map(u): smooth_out(u)
    sticky: 0

Lerping between variables that are meant to be integers wont leave the interpolated variable as an integer, but that doesn't matter a good amount of times. Take a look: 

func RootsOfUnity(N) = identity:
    let n = round(N)

    var dots = {}
    for i in 0 :< n
        let theta = TAU * i / n
        let c = cos(theta)
        let s = sin(theta)
        dots += Dot:
            point: {c,s,0}
            tag: {}
            dot: BLACK

    element: dots

tree static
tree main

let ring = Circle:
    center: ORIGIN
    radius: 1
    tag: {}
    stroke: RED

var roots_it = RootsOfUnity:
    N: 0

play += Show:
    into&: static
    value: ring
    sticky: 0
play += Show:
    into&: main
    value: roots_it
    sticky: 0

play += Lerp:
    dst&: main
    start: roots_it
    end: RootsOfUnity:
        N: 64

    time: 3
    unit_map(u): smooth_in(u)
    sticky: 0

7.4 Keyframe

This brings us to keyframe. In the case that we have many succesive lerps of the same variable, we can use keyframe. A sine example is written here.

func Sine(A, omega, phi) = ExplicitFunc:
    start: -PI
    stop: PI
    f(x): A * sin(omega * (x - phi))
    tag: {}
    stroke: RED

tree main
var sine = Sine:
    A: 1
    omega: 1
    phi: 0

play += Show:
    into&: main
    value: sine
    sticky: 0

var frames = {0: sine}
frames[1] = sine
frames[1].A = 2

frames[2] = frames[1]
frames[2].omega = 2

frames[4] = Sine:
    A: 2.1
    omega: 12
    phi: PI / 2

frames[7] = frames[4]
frames[7].A = 1
frames[7].omega = 1

play += Keyframe:
    dst&: main
    time_to_value: frames
    unit_map(u): smooth(u)
    sticky: 0

Good to know, but from experience it's generally used occasionally.

7.5 (Optional) Transform vs Lerp

Lastly, I want to talk about when to use Transform vs Lerp. Transform's syntax is nicer (since we dont need to specify start), but it gives less control. We use Transform whenever it looks nice, or when we're transforming from unrelated types, so that there's not anything that can be lerped. If we need more control about intermediate states, Lerp is the way to go.

REMARK Finally, notice that there is some inconvenience in the syntax once again. It definitely is really repetitive, but again this is something I'm very confident can be fixed with a language feature (maybe macros, or maybe something like cpp's std::exchange, i have very many ideas actually). Nevertheless, this is the majority of what you need to know for animations.

7.6 Monocurl Logo Animation

Also, we can now create the Monocurl logo animation! Lets break it down.

In fact, there isn't that much going on. It does involve a camera animation which we haven't explicitly seen, but it's the same general idea.

  • Track 1: "Monocurl" needs to be written
  • Track 2: The graph and color grid
  • Track 3: The camera motion

Track 1 is pretty easy, we simply just need to use a Write animation. Track 3 is also not that bad, we use CameraLerp to move the camera from its default position to some offset target. Note that lerp would work as well, but the rotation is slightly off.

Track 2 might seem a bit difficult, but really it's just two simple animations. We first have a blank ColorGrid, then we change its color, and then we add elevation to it via PointMapped. Both interpolations can be done via Transform. Note Track 2 and Track 3 should be in parallel.

Lets see the code!

/* helpers */
let step = 0.05
func q(x,y) = 1.2 * ((x-0.5)**2 + (y-0.5)**2)
func col(x,y) = identity:
    let val = q(x,-y)
    let colors = {0:WHITE,0.2:PURPLE,0.5:BLUE}
    let color = keyframe_lerp(colors, val)
    element: color


tree main
tree static
tree text

/* Monocurl */
var tex = Text:
    text: "Monocurl"
    scale: 1.5
    stroke: BLACK
    fill: BLACK
tex = Centered:
    mesh: tex
    at: {0,1,0}

/* Main Graph */
var grid = ColorGrid:
    x_min: 0
    x_max: 1
    y_min: -1
    y_max: 0
    x_step: step
    y_step: step
    tag: {}
    color_at(pos): BLACK
    stroke: BLACK

/* Axes */
let axis = Axis3d:
    center: {0,0,-0.01}
    pos_x_axis: {1,0,0}
    pos_y_axis: {0,-1,0}
    pos_z_axis: {0,0,1}
    x_unit: 1
    x_min: 0
    x_max: 1
    x_label: "x"
    y_unit: 1
    y_min: 0
    y_max: 1
    y_label: "y"
    z_unit: 1
    z_min: 0
    z_max: 1
    z_label: "z"
    grid: on
    tag: {}
    color: BLACK

var camera_it = camera

/* intro */
play += Write:
    into&: text
    value: tex
    time: 1
    sticky: 0
play += FadeIn:
    into&: static
    value: axis
    time: 1
    sticky: 1
play += FadeIn:
    into&: main
    value: grid
    time: 1
    sticky: 1

/* main animation */
var grid_anim = {}
grid_anim += Transform:
    grid = ColorGrid:
        x_min: 0
        x_max: 1
        y_min: -1
        y_max: 0
        x_step: step
        y_step: step
        tag: {}
        color_at(pos): col(pos[0], pos[1])
        stroke: BLACK
    mesh&: main
    target: grid
    null_action: fade
    time: 1
    sticky: 0
grid_anim += Transform:
    grid = PointMapped:
        mesh: grid
        point_map(point): {point[0], point[1], q(point[0], point[1] + 1)}
    mesh&: main
    target: grid
    null_action: fade
    time: 2
    sticky: 0
play += grid_anim

/* camera movement */
play += CameraLerp:
    var next = camera_it
    next.origin = {2,-2,1.4}
    next.up = {0,0,1}

    camera&: camera
    start: camera_it
    end: next
    time: 3
    sticky: 1

play += Wait(1, 0)

Notice that the main animation honestly isn't that complex. Hopefully you now feel like you could have come up with the sequence.
